# Migrating from Stainless to Inox

> **May 2026:** Anthropic acquired Stainless and is [winding down all hosted Stainless products](https://techcrunch.com/2026/05/18/anthropic-has-acquired-the-dev-tools-startup-used-by-openai-google-and-cloudflare/), including the SDK generator. New signups, projects, and SDKs are no longer available; existing customers keep the SDKs they've generated but need **local generation or another vendor** going forward.
>
> **Inox is local generation.** It runs entirely in your own infra, it's MIT-licensed, and it's free. This guide gets you off the hosted Stainless workflow.

---

## TL;DR

1. Export your last Stainless config + OpenAPI spec (you still have access to your generated repos).
2. Drop your `openapi.yaml` next to an `inox` config.
3. `npx @crevious/inox generate` → idiomatic SDKs for TypeScript, Python, Go, Java, Ruby, C#.
4. Wire it into CI (a workflow is generated for you). Done — no cloud, no per-SDK bill.

You keep owning your published SDKs; Inox just replaces the *generation* step that used to live in Stainless's cloud.

---

## What carries over (concept map)

| Stainless | Inox |
|---|---|
| `stainless.yml` (config) | `inox.yml` / `sdkgen.yml` |
| Hosted generation (cloud) | `inox generate` (your machine / CI) |
| Pagination config | `pagination:` block (cursor, cursor_id, cursor_url, offset, page_number + bidirectional) |
| Typed errors | per-status error classes generated by default |
| Auto-pagination iterators | `*AutoPaging` / `auto_paging` in every language |
| `withRawResponse` | `withRawResponse` / `with_raw_response` / `WithRawResponse` |
| Request IDs | `x-request-id` extracted onto responses + errors |
| Streaming (SSE/JSONL) | `streaming:` config, `Stream<T>` per language |
| Webhooks | Standard-Webhooks HMAC verification |
| OAuth2 | client-credentials + refresh + **auth-code + device flow** |
| Retries / timeouts / idempotency | built in, configurable |
| Custom code (`.stainless` overlays) | 3-way-merge custom-code overlay |
| Managed publish to npm/PyPI | generated release workflow (you own the tokens) |

If your Stainless setup used a feature not in this table, open an issue — most of it already exists or is a small config flag.

## Step 1 — get your spec

You still have read access to your Stainless-generated repos. The OpenAPI spec you fed Stainless is what Inox needs. If you only have the generated SDK, the spec is usually committed alongside it, or you can re-export it from your API framework.

## Step 2 — minimal config

```yaml
# inox.yml
sdkgen: 1
organization: { name: acme }
project: { name: acme-api, display_name: Acme API }
spec: { path: ./openapi.yaml }
targets:
  typescript: { package_name: "@acme/acme", edition: typescript.2026-05-25 }
  python: { package_name: acme, project_name: acme-sdk, edition: python.2026-05-25 }
  go: { module_path: github.com/acme/acme-go, edition: go.2026-05-25 }
  ruby: { gem_name: acme, edition: ruby.2026-05-25 }
  java: { namespace: com.acme, maven_group: com.acme, maven_artifact: acme-java, edition: java.2026-05-25 }
  csharp: { namespace: Acme, edition: csharp.2026-05-25 }
client:
  class_name: Acme
  env_prefix: ACME
  base_url: { default: https://api.acme.test }
  auth:
    bearer: { env: ACME_API_KEY }
```

## Step 3 — generate

```bash
npx @crevious/inox generate --config inox.yml --out ./generated
```

Six SDK folders, each with zero runtime dependencies.

## Step 4 — port pagination / auth / streaming

Mirror your Stainless config. Examples:

```yaml
pagination:
  list_cursor:
    type: cursor
    items: $.data
    request_cursor: cursor
    response_next_cursor: $.next_cursor
    # bidirectional (optional):
    request_prev_cursor: before
    response_prev_cursor: $.prev_cursor
```

```yaml
client:
  auth:
    bearer: { env: ACME_API_KEY, prefix: "Bearer" }   # custom prefix supported, e.g. "Token"
    oauth2:
      token_url: /oauth/token
      authorization_url: /oauth/authorize          # auth-code flow helpers
      device_authorization_url: /oauth/device      # device flow helpers
      client_id_env: ACME_CLIENT_ID
      client_secret_env: ACME_CLIENT_SECRET
```

## Step 5 — CI

Inox emits a release workflow per language. You hold the publish tokens (npm/PyPI/etc.) — generation runs in *your* CI, the spec never leaves it.

## Why teams are switching to Inox specifically

- **Local / air-gapped** — the exact thing the Stainless wind-down forces you to find. Your spec never leaves your infra.
- **MIT, forkable** — no second vendor lock-in. Read and change the generator.
- **Zero-dependency SDKs** — nothing to audit but the standard library.
- **Free** — no per-SDK subscription.
- **Verified on 22 real specs** (OpenAI, Stripe, GitHub, Cloudflare, Twilio, …).

## What's not at parity yet (be honest)

- Languages: 6 today (TS/Py/Go/Java/Ruby/C#). Kotlin/PHP not yet shipped.
- Inox is self-managed: there's no hosted dashboard or white-glove "we publish for you" service. That's the trade for owning the toolchain.
